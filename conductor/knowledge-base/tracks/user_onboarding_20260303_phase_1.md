# Track User & Identity Management - Onboarding: Phase 1 Knowledge Note

## The Enterprise Challenge
Managing complex identity states and onboarding flows in a Zero-Trust FinTech environment requires absolute data integrity. Implementing a strict State Machine at the core Domain layer prevents invalid state transitions (like logging in before approval) before they can even reach the database or application layer.

## Knowledge Hierarchy

### Junior Level (The "What")
- **Domain Driven Design (DDD) Entity:** The `ApplicationUser` class is our Aggregate Root for users. It encapsulates not just the user's data (like `Email`), but also the business rules surrounding that data.
- **State Machine:** We introduced a `UserStatus` enum (`Created`, `PendingApproval`, `PendingVerification`, `Active`) and methods like `StartCustomerOnboarding` that strictly control how a user moves between these states.
- **Domain Events:** We created `IDomainEvent` markers like `UserRegisteredEvent`. These act as "messages" generated by our Domain when an important change happens, which the outer layers can later respond to.

### Mid Level (The "How")
- **Encapsulation:** The `Status` property has a `private set;`, meaning it can only be changed by invoking the specific behavioral methods (e.g., `ApproveAccount`). This guarantees no developer can accidentally bypass the onboarding flow by setting `Status = UserStatus.Active` arbitrarily in an API endpoint.
- **Red-Green-Refactor:** The implementation was guided entirely by xUnit tests (`ApplicationUserTests`), ensuring every single state transition scenario (including invalid ones) was empirically proven before moving forward.
- **EF Core Integration:** By updating the entity and generating an EF Core migration (`AddUserStatus`), we ensured the new schema state correctly mapped down to our PostgreSQL persistence without breaking the domain's purity.

### Senior/Principal Level (The "Why")
- **"Fail Fast" Security Posture:** By enforcing invariant rules at the instantiation and mutation levels of the Aggregate Root, we achieve mathematical certainty that the system cannot enter an invalid state. A user in `PendingApproval` cannot physically be "Logged In" because the `CanLogin()` method is derived directly from the immutable `Status` state.
- **Event-Driven Decoupling:** Instead of making the `ApplicationUser` responsible for sending a welcome email or a SignalR notification directly, we emit a `UserRegisteredEvent`. This allows the Domain layer to remain pure and completely oblivious to infrastructure concerns like Service Buses, fulfilling the strict Clean Architecture requirement.

## Deep-Dive Mechanics
The `SeedData.cs` incident highlighted an important integration dynamic. When using PostgreSQL advisory locks (`pg_advisory_lock`) in local dev environments, the connection to the specific database must be established *before* `EnsureCreated` or `Migrate` is called. If the database schema isn't fully synched (or dropped), this locking strategy will fail. Thus, maintaining synchronized EF Core migrations is critical for smooth deterministic bootstrapping.

## Interview Talking Points (Tiered)
- **Junior/Mid:** "I implemented a strict State Machine within the `ApplicationUser` entity using encapsulated methods and private setters. This ensures that users can only be activated through the proper onboarding flow, which I validated extensively using xUnit."
- **Senior/Lead:** "To satisfy the Zero-Trust mandate, I shifted the authorization invariant into the core Domain layer. By using Domain Events for state transitions, we paved the way for an Event-Driven architecture where security notifications (like SignalR pushes) can be triggered entirely as side-effects, keeping the core system perfectly decoupled from infrastructure mechanisms."

## March 2026 Market Context
In 2026, FinTech architecture eschews "Anemic Domain Models" (where entities are just bags of getters and setters manipulated by external services). By implementing "Rich Domain Models" using C# 14 features, we build a system where the business logic is the code itself. This is the "Gold Standard" for passing high-stakes security audits like SOC 2 or FAPI 2.0, as it mathematically minimizes the risk of logical bypass vulnerabilities.
